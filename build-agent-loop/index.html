01

What it does

Most of us use Claude for one-off tasks. But a second pattern quietly eats hours every week — the recurring check: the daily pipeline scan, the weekly progress report, the nightly log review. Work that isn't hard, just relentless.

Build Agent Loop is a Claude Code skill that sets those up once, correctly, with real guardrails. When you ask Claude to build a loop, the skill walks it through qualifying the task, scaffolding a tiny workspace, baking in verification from day one, and only scheduling once it's proven stable.

The design goal was never to give Claude more authority — it's to take the boring, repetitive middle of a task off your plate so you get the time back.

02

The four parts of a good loop

Designing a loop is really four choices. Most people nail the first three and skip the fourth — the one that decides whether the loop can ever run unattended.

01

Trigger

What wakes the loop, and how cheap that wake-up is. Prefer the cheapest trigger that still catches the event.

02

File structure & contract

Where the goal, the memory, and the outputs live. The single most important decision — a loop is only as good as the state it can read back.

03

Tools & connectors

What it's allowed to touch, as explicit allow/deny rules for files, commands, and connectors — never a vague "be careful."

Everyone skips
04

Self-verifying workspace

Objective gates — tests, linters, required files, exact-count checks — that let the loop grade its own output.

03

Serious loops split into three roles

A trivial loop does everything in one pass. A serious one separates three roles so no single step gets to mark its own homework as correct.

Plan

Orchestrator

Reads the memory and the contract, decides what this run should do, and dispatches it. Usually cheap.

Act

Worker

Performs the one bounded action in scope. It does the work — it does not get to decide whether the work is good.

Check

Evaluator

Independently verifies the output against the checklist and writes the lesson into memory. Never the same step that produced the output.

Orchestrator plans → Worker acts → Evaluator checks & remembers → repeat

04

Memory that compounds, a loop that evolves

The difference between a loop and a prompt on a timer is that a loop remembers — and gets better on purpose instead of staying as dumb as day one.

PROGRESS.md + MEMORY.md

Carry the context forward

PROGRESS.md is short-term working memory: status, last run, open items, and a "do not repeat" list. MEMORY.md is long-term distilled memory: durable decisions, cached facts, reusable snippets, and the evolution log.

Every run reads both before acting and writes both before stopping. That's the one thing that separates a loop from re-typing a prompt.

The evolution pass

Every run compiles a cheaper next run

Each run leaves behind cached facts and reusable scripts, so mechanical work runs deterministically and for free and the model is saved for the genuinely hard part.

Every N runs a self-improvement pass rewrites the loop itself — tighter contract, cheaper trigger, dead instructions pruned. It's verified like any run, and it may only tighten the safety boundary; loosening always needs a human.

06

The permission ladder

Loops start at the bottom and only move up after the current level has run reliably. Nothing sensitive happens without a human clicking the button.

  • L1 Read-only analysis
  • L2 Draft output to outputs/
  • L3 Sandbox / branch edits only
  • L4 Draft external actions (draft PR or message — not sent)
  • L5 Human-approved writes
  • L6 Fully automated low-risk actions (narrow, proven cases only)
