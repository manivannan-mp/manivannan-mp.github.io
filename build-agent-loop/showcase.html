Agent Loops: 13 Role-Based Use Cases

A loop is a repeatable workflow wrapped around Claude: something triggers a run, Claude reads the memory of prior runs, takes a constrained action, checks its own output, saves what it learned, and decides whether to stop, retry, or escalate to a human. The durable spine of memory + contract is what turns re-typing a prompt into a loop that compounds. Every loop below starts read-only or draft-only — the human keeps the click that matters.

The four parts of a good loop

Designing a loop is really four choices. Most people nail the first three and skip the fourth — the one that decides whether the loop can ever run unattended.

01

Trigger

What wakes the loop, and how cheap that wake-up is. Prefer the cheapest trigger that still catches the event — a condition beats a tight timer that mostly finds nothing.

02

File structure & contract

Where the goal, the memory, and the outputs live. The single most important decision — a loop is only as good as the state it can read back.

03

Tools & connectors

What it's actually allowed to touch, as explicit allow/deny rules for files, commands, and connectors — never a vague "be careful."

Everyone skips
04

Self-verifying workspace

Objective gates — tests, linters, required files, exact-count checks — that let the loop grade its own output. Without this it can never run unwatched.

03

Serious loops split into three roles

A trivial loop does everything in one pass. A serious one separates three roles so no single step gets to mark its own homework as correct.

Plan

Orchestrator

Reads the memory and the contract, decides what this run should do, and dispatches it. Usually cheap.

Act

Worker

Performs the one bounded action in scope. It does the work — it does not get to decide whether the work is good.

Check

Evaluator

Independently verifies the output against the checklist and writes the lesson into memory. Never the same step that produced the output.

Orchestrator plans → Worker acts → Evaluator checks & remembers → repeat

04

Carry the memory forward

A loop that forgets is just a prompt you retype. Two files hold the memory — every run reads both before acting and writes both before stopping. This is the one thing that separates a loop from a re-run.

PROGRESS.md

Short-term working memory

Current status, last-run summary, open items, blockers, what the next run should do, and a "do not repeat" list. Kept compact.

MEMORY.md

Long-term distilled memory

Durable decisions and why, cached facts (IDs, thresholds, gotchas), reusable snippets, and the evolution log. The asset that makes each run cheaper than the last.

05

Evolve, or stay as dumb as day one

Every run compiles a cheaper, smarter next run

Treat each run as compiling assets for the next: cached facts so it doesn't re-derive them, reusable scripts so mechanical work runs deterministically and for free, a sharper "do not repeat" list, and a tighter contract.

Push the boring, certain part into a deterministic pre-stage and reserve the expensive model for the genuinely hard, judgment-heavy part. A loop that gets more expensive per run is overdue for evolution.

A periodic pass that rewrites the loop itself

Every N runs, run an evolution pass instead of a normal run: tighten anything ambiguous, make the trigger cheaper, fold repeated steps into scripts, prune dead instructions, and promote hard-won facts into MEMORY.md.

Two hard rules: the evolution pass is verified like any other run, and it may only tighten the safety boundary automatically — loosening scope or raising the permission level always needs a human.

06

Pick your loop

Before you schedule anything: qualify the task (it repeats on a real cadence, success is objectively checkable, and future runs benefit from memory), then run the loop by hand 3–5 times and watch what it touches. These are starting templates, not proof a loop is ready to run unsupervised. Keep the permission level low, prefer draft-not-act, and never hand a loop unsupervised authority over money, contracts, grades, live production systems, or messages sent on your behalf.
