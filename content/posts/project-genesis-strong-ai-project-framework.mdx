## Why Strong Projects Begin Before Coding

Most products start with a feature request. Exceptional products begin with a clear problem, a durable vision, explicit boundaries, and a shared operating system. The first five percent determines the next ninety-five.

## Two Stages. One Builder.

Do not put Codex and Claude Code in the same critical path by default. Pick one implementation engine and let it own delivery from sprint to release.

<div className="post-stage-grid">
  <article className="post-stage-card">
    <p className="post-kicker">Stage 1</p>
    <h3>ChatGPT: Genesis Engine</h3>
    <p>Clarifies the problem, challenges assumptions, defines the product vision, architecture, non-negotiables, roadmap, and ProjectOS.</p>
    <p><strong>Focus:</strong> Vision, architecture, ProjectOS.</p>
  </article>
  <article className="post-stage-card">
    <p className="post-kicker">Stage 2A</p>
    <h3>Codex: Implementation Engine</h3>
    <p>Reads the Project Bible, executes sprints, writes code, runs tests, fixes failures, updates state, and prepares releases.</p>

  </article>
  <article className="post-stage-card">
    <p className="post-kicker">Stage 2B</p>

    <p>Uses the same operating model as Codex. Choose it instead of Codex when it is the better fit for your repository and workflow.</p>
    <p><strong>Focus:</strong> Alternative implementation engine, not parallel by default.</p>
  </article>

<div className="post-section-card">

### The Golden Rules

1. Vision before implementation.
2. Architecture before code.
3. One source of truth.
4. One implementation engine per project.
5. Every sprint ends with working, verified output.
6. Decisions and project memory are updated continuously.

</div>

> **The important correction:** Codex and Claude Code are not separate permanent roles. They are interchangeable implementation engines. A project should normally choose one, not require both.

Independent review can still be requested later, but it should be an optional quality gate, not a dependency that slows every decision.

## The Operating Workflow

The framework is simple enough to repeat and strong enough to scale from a weekend prototype to a multi-year platform.

<div className="post-section-card">

### 1. Start With the Idea

Describe the problem, who has it, why it matters, and what you currently believe the solution could be.

### 2. Run Project Genesis in ChatGPT

Challenge the idea, establish the vision, research the space, define principles, and create the architecture.

### 3. Generate the Project Bible

Create the durable handoff: memory, constitution, decisions, contracts, quality gates, roadmap, milestones, and sprint system.

### 4. Choose One Implementation Engine

Select Codex or Claude Code for the repository. Do not require both by default.

### 5. Execute One Verified Sprint at a Time

Build, test, validate, repair, document, and update state before moving forward.

### 6. Release Through Milestones

Ship internal prototype, private alpha, public beta, and production v1 with explicit quality bars for each.

## Use This as Your Opening Message

A strong project starts with a strong opening.

Do not begin with "build this feature." Begin by asking the AI to challenge the idea, define the product, establish the rules, and create the implementation handoff.

<PromptBlock
  kicker="Opening prompt"
  title="Project Opening Prompt"
  text={`I want to start a new project.

Project idea:
[Describe the idea]

Target users:
[Who has this problem?]

Problem:
[What is painful, slow, expensive, risky or impossible today?]

My current solution idea:
[Describe your early thinking]

Constraints:
[Budget, time, technology, geography, team, compliance or other limits]

Do not jump directly into coding.

Use the Project Genesis Framework:
1. Challenge the idea from first principles.
2. Clarify the real problem.
3. Research the current landscape where needed.
4. Define the product vision, principles, users, value proposition and non-goals.
5. Design a scalable but practical architecture.
6. Identify risks, dependencies and trade-offs.
7. Create the Project Bible / ProjectOS.
8. Create an implementation-ready handoff for either Codex or Claude Code.
9. Define prototype, alpha, beta and v1 release milestones.
10. End with the exact first implementation prompt.

Treat this as the beginning of an exceptional product, not a one-off feature.`}
